# 日常使用工作流

## 工作流一：读项目

适合刚接手项目或准备做修改。

提示词：

```text
请阅读 README、配置文件和主要源码目录，回答：
1. 这个项目解决什么问题？
2. 主要模块分别负责什么？
3. 本地如何启动和测试？
4. 你认为我改动前需要注意哪些风险？
暂时不要修改文件。
```

预期输出：

- 项目地图。
- 关键入口。
- 启动命令。
- 风险提醒。

## 工作流二：小范围修复

适合 bug fix、文档修正、脚本小改动。

提示词：

```text
请修复这个问题：运行订单导出时，如果 end_date 为空会报错。
只允许修改 src/export_order.py 和 tests/test_export_order.py。
改完后运行 pytest tests/test_export_order.py。
```

关键点：

- 明确问题。
- 限定文件范围。
- 给出验证命令。
- 要求总结变更。

## 工作流三：生成测试

适合遗留代码质量提升。

提示词：

```text
请为 src/metric_parser.py 补充单元测试。
优先覆盖正常输入、空输入、非法字段、重复字段这四类情况。
不要修改业务逻辑，除非你发现明确 bug，先向我说明。
```

验收标准：

- 测试命名清晰。
- 覆盖关键边界。
- 不引入真实敏感数据。
- 能在本地稳定运行。

## 工作流四：代码审查

适合提交前自查。

提示词：

```text
请以 code review 的方式检查当前 git diff。
优先指出 bug、回归风险、缺少测试和安全问题。
不要直接修改文件，先列出 findings。
```

建议让 Codex 先 review，再决定是否让它修复。

## 工作流五：文档沉淀

适合把经验变成知识库。

提示词：

```text
请根据当前目录里的材料，整理一份内部分享教程。
要求：
- 面向第一次使用的人。
- 包含安装、配置、验证和常见问题。
- 使用中文。
- 输出为 Markdown。
```

适合沉淀：

- 安装教程。
- 项目 onboarding。
- 业务指标说明。
- 故障排查 SOP。
- 分享大纲。

## 工作流六：SQL 和数据脚本检查

适合数据团队。

提示词：

```text
请检查这个 SQL 是否存在口径风险。
重点看：
- 时间过滤是否一致。
- join 是否可能放大数据。
- null 处理是否合理。
- 指标命名是否容易误解。
不要连接数据库，只基于 SQL 文本分析。
```

注意：

- 不要粘贴客户明细。
- 不要暴露生产连接串。
- 复杂业务口径仍需业务 owner 确认。

## 工作流七：让 Codex 帮你跑验证

提示词：

```text
请运行项目已有的测试命令。
如果测试失败，先分析失败原因，不要立刻大范围修改。
```

常见验证命令：

```powershell
npm test
npm run lint
pytest
dotnet test
git diff --check
```

## 好提示词模板

```text
背景：
目标：
范围：
不要做：
验证方式：
输出格式：
```

示例：

```text
背景：这个仓库是团队内部知识库，主要是 Markdown 分享材料。
目标：补齐 Codex 在中国大陆安装使用教程。
范围：只修改 Codex安装和使用分享 目录，必要时更新根 README。
不要做：不要删除其他分享资料，不要加入真实账号信息。
验证方式：检查 Markdown 链接和文件列表。
输出格式：总结新增文件和建议阅读顺序。
```

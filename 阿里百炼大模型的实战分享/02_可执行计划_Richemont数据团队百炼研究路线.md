# 可执行计划：Richemont 数据团队百炼研究路线

## 1. 总体目标

在 8 到 10 周内完成一套可复用的阿里百炼最佳实践资产，并完成至少 1 个可演示 POC。

最终交付：

- 一套百炼与阿里 AI 生态知识材料。
- 一套数据团队 AI 应用开发规范。
- 一个 Richemont Data Knowledge Assistant POC。
- 一套 AI 应用评测与上线前检查模板。
- 一场面向团队或管理层的 Demo 分享。

## 2. 成功标准

| 维度 | 验收标准 |
| --- | --- |
| 知识沉淀 | 至少形成 5 份可复用 Markdown 文档 |
| 技术验证 | 完成模型 API 调用、知识库问答、引用来源展示 |
| 业务验证 | 至少覆盖 30 个真实数据/报表/指标问题 |
| 效果验证 | 标准问题集准确率达到 80% 以上，严重幻觉为 0 |
| 治理验证 | 形成权限、敏感信息、成本、人工确认的检查清单 |
| 分享验证 | 完成 1 次 30 到 45 分钟内部分享或管理层展示 |

## 3. 阶段路线

### 第 0 周：准备和边界确认

目标：

- 明确研究范围和 POC 主题。
- 确认百炼账号、地域、权限、API Key、预算。
- 确认哪些 Richemont 内部资料可以用于 POC。

任务：

- 申请或确认阿里云百炼访问权限。
- 建立资料清单，包括数据字典、报表说明、FAQ、SOP。
- 确认资料分级：公开内部资料、敏感资料、不能进入 POC 的资料。
- 明确 POC 使用范围：仅内部演示，还是可开放给业务试用。

交付物：

- POC 范围说明。
- 数据资料清单。
- 风险与权限确认表。

### 第 1 周：平台入门与 API 验证

目标：

- 跑通百炼模型调用。
- 理解 DashScope 与 OpenAI 兼容接口。
- 建立团队最小代码样例。

任务：

- 完成 API Key 配置和本地环境变量管理。
- 调用 1 个通用对话模型。
- 调用 1 个 Embedding 模型。
- 记录模型参数：temperature、top_p、max_tokens、stream。
- 记录错误处理：鉴权失败、限流、超时、重试。

交付物：

- `model_call_demo.py` 或等价样例。
- 百炼 API 接入笔记。
- 团队模型选择初版。

### 第 2 周：RAG 知识库最小闭环

目标：

- 用少量真实资料完成知识问答闭环。
- 让答案带引用来源。

任务：

- 选择 10 到 20 份低敏资料。
- 清洗文档并统一命名规则。
- 建立知识库或自建向量检索样例。
- 准备 20 个标准问题。
- 验证答案是否基于资料回答。

交付物：

- 知识库配置记录。
- 标准问题集 V1。
- RAG 问答效果记录。

### 第 3 周：知识库质量优化

目标：

- 优化切片、元数据、召回和提示词。
- 建立可重复的 RAG 调优方法。

任务：

- 对比不同 chunk size 和 overlap。
- 加入元数据：业务域、报表名、指标名、更新时间、资料 owner。
- 设计拒答策略：资料中没有就明确说明。
- 记录 TopK、相似度阈值对答案质量的影响。
- 对标准问题集做人工评分。

交付物：

- RAG 调优记录。
- Prompt 模板 V1。
- 评测结果表 V1。

### 第 4 周：Data Knowledge Assistant POC

目标：

- 做出可演示的知识助手。
- 支持业务常见问题和数据团队问题。

任务：

- 设计用户入口，可以是百炼应用、简单 Web 页面、Notebook 或命令行 Demo。
- 支持问题输入、答案输出、引用来源、无法回答提示。
- 加入使用边界提醒：答案仅供内部参考，关键结论需人工确认。
- 准备 Demo 脚本和演示问题。

交付物：

- 可运行 POC。
- Demo 问题清单。
- 演示脚本。

### 第 5 周：数据分析 Copilot 探索

目标：

- 验证大模型辅助数据分析的可行性。
- 不直接接入生产库写操作。

任务：

- 整理 10 个典型分析问题。
- 让模型生成 SQL 草稿或分析思路。
- 由数据分析师人工校验 SQL。
- 研究与 DataWorks、MaxCompute、Hologres、Quick BI 的结合方式。
- 识别需要权限控制的风险点。

交付物：

- SQL 生成样例。
- 数据分析 Copilot 场景评估。
- 风险清单。

### 第 6 周：工作流和智能体探索

目标：

- 找到适合工作流或智能体的可控流程。

任务：

- 选择一个流程：活动复盘、报表异常解释、数据质量排查。
- 拆解流程节点：输入、检索、判断、生成、人工确认、输出。
- 设计工具调用边界。
- 记录哪些环节必须人工确认。

交付物：

- 工作流设计图。
- 智能体边界说明。
- 流程 Demo 或原型。

### 第 7 周：评测、治理与成本控制

目标：

- 把 Demo 升级为可治理的 POC。

任务：

- 扩充标准问题集到 30 到 50 个。
- 评估准确率、引用命中率、拒答率、幻觉率。
- 记录响应延迟和 Token 成本。
- 明确敏感信息处理方式。
- 建立上线前检查清单。

交付物：

- 评测报告。
- 成本测算。
- 上线前检查清单。

### 第 8 周：内部分享与下一阶段规划

目标：

- 完成知识分享。
- 为下一阶段争取试点或资源。

任务：

- 准备 30 到 45 分钟分享材料。
- 演示 Data Knowledge Assistant。
- 展示评测结果、风险控制和下一步路线。
- 收集团队和业务反馈。

交付物：

- 分享大纲。
- Demo 视频或截图。
- 下一阶段 Roadmap。

## 4. 角色分工建议

| 角色 | 主要职责 |
| --- | --- |
| 项目 Owner | 范围管理、节奏推进、管理层同步 |
| AI 技术负责人 | 百炼接入、RAG、智能体、Demo 实现 |
| 数据领域专家 | 指标口径、报表知识、标准问题集 |
| 业务代表 | 提供真实问题、参与验收 |
| 治理负责人 | 权限、敏感信息、合规、成本 |

如果团队人少，可以一人兼任多个角色，但至少要明确“技术实现”和“业务验收”不是同一个视角。

## 5. 每周例会模板

- 本周完成了什么？
- 哪些假设被验证或推翻？
- 当前最大风险是什么？
- 下周要交付什么可见成果？
- 是否需要管理层或业务方决策？

## 6. 下一阶段扩展方向

完成第一轮 POC 后，可以从以下方向扩展：

- 从数据知识助手扩展到报表解释助手。
- 从只读问答扩展到半自动报告生成。
- 从单一团队资料扩展到跨部门知识库。
- 从人工上传资料扩展到自动同步文档源。
- 从简单评测扩展到持续评测和线上反馈闭环。
